# Interview-Leitfaden

Katalog, kein Skript: Formulierungen an Pitch und bisherige Antworten anpassen.
Bereits beantwortete Fragen nie erneut stellen.

## Runde A — Kern
1. Welches konkrete Problem löst das? Woran merkt man das Problem heute?
2. Wer genau hat das Problem (Rolle, Situation)? Wer ist der erste echte Nutzer?
3. Was ist der eine Satz, der die Idee trägt? (Vorschlag anbieten, bestätigen lassen)
4. Woran erkennst du in 3 Monaten, dass es funktioniert? (Erfolgskriterium)

## Runde B — Scope
1. Was muss die allererste nutzbare Version zwingend können? (→ 3–7 Muss-Ergebnisse, prüfbar formulieren)
2. Was gehört ausdrücklich NICHT dazu — auch wenn es naheliegt? (→ mind. 3 Nicht-Ziele; Kandidaten aktiv anbieten)
3. Gibt es eine noch kleinere Version, die schon Wert liefert? (MVP-Schnitt testen)
4. Was existiert schon (Code, Daten, Accounts, Designs)?

## Runde C — Rahmen
1. Zeithorizont: Wann soll etwas Nutzbares existieren?
2. Kosten: Dürfen kostenpflichtige Dienste/APIs genutzt werden? Bis zu welchem Rahmen?
3. Tech-Vorgaben: bestimmte Sprache/Plattform/Hosting — oder entscheidet der Agent? („Agent entscheidet" ist eine valide, explizit festzuhaltende Antwort)
4. Zielumgebung: Web, Mobile, CLI, Desktop? Wo läuft es (lokal, Server, Cloud)?

## Runde D — Autonomievertrag
1. Standard-Eskalationsregeln vorlesen (Brief-Template) — passt das, oder strenger/lockerer?
2. Erweiterungsbudget: Wie viele Zusatzideen pro Milestone dürfen maximal aktiviert werden? (Default: 1)
3. Darf der Agent deployen/veröffentlichen, oder endet Autonomie an lokaler Lauffähigkeit?
4. Milestone-Verhalten: nur Report und weiterarbeiten (Default) — oder Stopp je Milestone?

## Definition of Ready
Erst loslegen, wenn jeder Punkt beantwortet oder als bestätigter Default gedeckt ist:

- [ ] Kernidee in einem Satz, vom User bestätigt
- [ ] Zielnutzer und Kernproblem klar
- [ ] 3–7 prüfbare Muss-Ergebnisse
- [ ] mind. 3 explizite Nicht-Ziele
- [ ] Zeithorizont und Kostenrahmen
- [ ] Tech-Vorgaben oder explizit „Agent entscheidet"
- [ ] Autonomievertrag inkl. Deployment-Grenze bestätigt
- [ ] Erweiterungsbudget festgelegt
- [ ] Erfolgskriterium aus User-Sicht

## Umgang mit Antworttypen
- **Vage Antwort** („soll halt einfach sein"): in prüfbare Form übersetzen und Übersetzung bestätigen lassen.
- **Widerspruch** (z. B. Muss-Ergebnis kollidiert mit Nicht-Ziel): sofort benennen, User entscheiden lassen.
- **Featureliste statt Kern**: fragen „Wenn nur EINES davon ginge — welches?" und den Rest als Nicht-Ziel- oder IDEAS-Kandidaten vorschlagen.
